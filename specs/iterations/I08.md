## Objective

Transform mapx from a flat file graph into a **structured, navigable code intelligence system** with two complementary capabilities:

1. **Cluster intelligence** — infer the logical modules and domains of the codebase from namespace declarations, directory structure, and import density. Render these as nested DOT subgraphs and cluster-level summaries, enabling top-down and bottom-up navigation of any project.

2. **Data flow tracing** — trace how data propagates through the codebase along call chains, instantiations, and parameter flows. Identify entry points, terminal consumers, and circular paths. Answer "where does this request go?" and "what feeds into this service?".

After I08, mapx understands not just _what_ the code contains but _how it is organised_ and _how data moves through it_.

## F14 Summary: Cluster Detection

**What it adds**:
- `src/core/cluster-engine.ts` — `ClusterEngine` class
- Schema v4 migration: `clusters` and `cluster_membership` tables
- Three cluster sources: namespace declarations (PHP, TS), directory structure, edge-density community detection (Label Propagation)
- `ParseResult.fileMetadata` to carry namespace info from parsers
- PHP tree-sitter query addition: `namespace_definition` capture
- `ScanPhase` extended with `'cluster'` phase
- `mapx clusters` CLI command + `mapx_clusters` MCP tool

**Key design decisions**:
- Priority order: namespace > directory > community
- Files get exactly one primary cluster assignment
- Cluster depth is driven by directory nesting (or namespace segment count for namespace clusters)
- Community detection uses Label Propagation — O(E), fast, good enough

## F15 Summary: Cluster-Aware Export

**What it adds**:
- `src/exporters/cluster-dot-exporter.ts` — `ClusterDotExporter` class
- New `--cluster` and `--depth` flags on `mapx export`
- DOT `subgraph cluster_*` rendering for file-level cluster view
- DOT cluster-only view (`--depth=0`) with aggregate edge counts
- Fallback SVG renderer updated to draw cluster bounding boxes
- LLM exporter `## Structure` section (cluster hierarchy + inter-cluster edge counts)
- JSON export extended with `clusters` array

**Key design decisions**:
- Default `--cluster none` preserves exact backward compatibility
- Cluster colours are deterministically derived from name hash (consistent across re-runs)
- Edge `penwidth` proportional to `log(edgeCount) + 1` (encodes volume without overwhelming thin edges)

## F16 Summary: Data Flow Tracing

**What it adds**:
- `src/core/flow-tracer.ts` — `FlowTracer` class
- `mapx trace <symbol|file>` CLI command (text, DOT, JSON output)
- `mapx trace --sources` — list all entry points
- `mapx trace --sinks` — list all terminal consumers
- `mapx trace A --to B` — shortest data path between two points
- Cycle detection with path reporting
- `mapx_trace`, `mapx_sources`, `mapx_sinks` MCP tools

**Key design decisions**:
- Data-bearing edges: `call`, `instantiation`, `param_type`, `return_type`, `relation`, `dispatch`, `notify`, `route`
- Structural edges (`import`, `extends`, `implements`, `binding`) excluded by default; opt-in with `--include-structural`
- DFS with visited-on-current-path cycle detection (not just visited-globally)
- DOT trace output uses diamond/octagon/box/ellipse node shapes to encode start/sink/intermediate/source roles

## Recommended implementation order

1. **F14 first** — establishes the database schema and `ClusterEngine`. Everything else builds on stable data.
2. **F16 in parallel with F14** — `FlowTracer` reads from existing `edges` table; no dependency on F14 schema.
3. **F15 last** — requires F14 cluster data to be populated before cluster export can render anything meaningful.

## Notes

- F14's community detection is the most algorithmically novel part. If Label Propagation proves insufficient (poor cluster quality on real projects), fall back to greedy modularity maximisation using the graphology-communities package or a simple SCC (strongly connected components) algorithm.
- F16 is deliberately scoped to **edge-level traversal**, not symbol-level taint analysis. This is intentional — it gives high-value results (which files/classes are in the data path) without the complexity of SSA/PDG construction. Full taint analysis is a future feature.
- The `--to` flag in `mapx trace` uses BFS (not DFS) for shortest-path semantics. DFS is used for the general trace to maximise discovery of all reachable nodes.
