## Objective

Give mapx first-class awareness of web frameworks. After I13, a scan of a Django, Express, Rails, Spring, or Next.js project produces route edges that link URL paths to the handlers that serve them — enabling LLM context tools to answer questions like "which function handles `POST /users`?" or "what middleware runs before the admin routes?".

18 frameworks are supported across 9 language ecosystems: Python (Django, Flask, FastAPI), JavaScript/TypeScript (Express, NestJS), frontend (React Router, Tanstack Router, Next.js, SvelteKit), PHP (Laravel extended, Drupal, Symfony, Yii2/Yii3, WordPress), Ruby (Rails), Java (Spring Boot), Go (Gin, chi, gorilla/mux), Rust (Axum, actix-web, Rocket), C# (ASP.NET Core), and Swift (Vapor). 21 frameworks total.

## Implementation phases

F21 is a prerequisite for all other features — implement and merge it before starting F22–F25. F22–F25 are independent of each other once F21 is available.

### Phase 1: Framework infrastructure (F21)

This phase establishes the shared foundation.

1. **New `ReferenceType` values** in `src/types.ts`:
   - `hook` — Drupal hook implementations
   - `graphql_resolver` — NestJS GraphQL operations
   - `message_handler` — NestJS microservice message/event patterns
   - `websocket_handler` — NestJS WebSocket subscriptions
   (The `route` and `middleware` types were introduced in F08 specs; implement them here if not yet in codebase.)

2. **Schema v6 migration** in `src/core/store.ts` / `store-bun.ts` / `store-node.ts`:
   - `ALTER TABLE edges ADD COLUMN metadata TEXT` (JSON, nullable)
   - Automatic migration: check `user_version` at startup, apply if `< 6`
   - Existing rows remain unaffected (NULL metadata)

3. **`FrameworkDetector` interface** in `src/core/scanner.ts` (or new `src/core/framework-scanner.ts`):
   ```typescript
   interface FrameworkDetector {
     name: string;
     detect(projectRoot: string, files: string[]): Promise<boolean>;
     extractRoutes(file: string, content: string, ctx: ScanContext): Promise<RouteBinding[]>;
     filePattern: RegExp;
   }
   ```

4. **`FrameworkRegistry`** singleton — holds all registered detectors.

5. **`RouteRegistry`** — in-memory store of all extracted `RouteBinding` objects; serialized to `.mapx/routes.json` after each scan.

6. **Scanner Phase 2** — after standard parse, run active framework detectors on matching files. Results written to `edges` table with `metadata` JSON column.

7. **Detection caching** — write detected framework names to `.mapx/frameworks.json`. Cache invalidated when `package.json`, `go.mod`, `Cargo.toml`, `requirements.txt`, `Gemfile`, or `pom.xml` changes.

8. **`mapx routes` CLI command**:
   - Reads from `RouteRegistry` / `.mapx/routes.json`
   - Options: `--framework=<name>`, `--method=<GET|POST|...>`, `--path=<pattern>`, `--json`

9. **`mapx_routes` MCP tool** — same data as CLI, returned as JSON for LLM consumption.

10. **F08 refactor** — move existing Laravel route extractor from `src/parsers/languages/php.ts` into `src/frameworks/detectors/laravel.ts` implementing `FrameworkDetector`.

### Phase 2: Python frameworks (F22)

Implement `src/frameworks/detectors/django.ts`, `flask.ts`, `fastapi.ts`.

1. Django `urlpatterns` extraction (regex over `urls.py`)
2. DRF router `register()` expansion to 6 CRUD edges
3. Flask `@app.route()` and `@blueprint.route()` + blueprint prefix resolution
4. Flask-RESTX `api.add_resource()` expansion
5. FastAPI `@app.get/post/...()` and `APIRouter(prefix=...)`

### Phase 3: Node.js/TypeScript frameworks (F23)

Implement `src/frameworks/detectors/express.ts` and `nestjs.ts`.

1. Express `app.METHOD("path", ...middlewares, handler)` — route + middleware edges
2. Express `router.route("path").get(h).post(h)` chain
3. Express `app.use("prefix", router)` prefix resolution
4. NestJS `@Controller` + `@Get/@Post/...` → HTTP route edges
5. NestJS `@Resolver` + `@Query/@Mutation/@Subscription` → `graphql_resolver` edges
6. NestJS `@MessagePattern` / `@EventPattern` → `message_handler` edges
7. NestJS `@SubscribeMessage` → `websocket_handler` edges
8. NestJS `@UseGuards` → `middleware` edges to guard classes

### Phase 4: Frontend routing (F24)

Implement `src/frameworks/detectors/react-router.ts`, `tanstack-router.ts`, `nextjs.ts`, `sveltekit.ts`.

1. React Router JSX `<Route path element>` tree traversal
2. React Router `createBrowserRouter` object-array form
3. Tanstack Router file-based routing (directory walk)
4. Tanstack Router `createRoute({ path, component })` code-based
5. Next.js App Router: `page.tsx` → page routes, `route.ts` → API routes
6. Next.js Pages Router: `pages/` and `pages/api/` filesystem routing
7. Next.js `'use server'` function detection
8. SvelteKit `+page.svelte`, `+page.server.ts`, `+server.ts` conventions

### Phase 5: Backend frameworks (F25)

Implement detectors for remaining 9 frameworks.

1. Laravel extended (invokable controllers, route groups, `->name()`, `apiResource`, subdomain)
2. Drupal: routing YAML parser + hook pattern scanner
3. Rails: `config/routes.rb` parser (verbs, `resources`, `namespace`, `scope`)
4. Spring Boot: `@RestController`, `@RequestMapping`, HTTP method annotations
5. Gin: `r.GET/POST/...("path", handler)` + `r.Group`
6. chi: `r.Get/Post/...("path", handler)` + `r.Route`
7. gorilla/mux: `r.HandleFunc("path", handler).Methods(...)`
8. Axum: `.route("path", get(handler).post(handler))`
9. actix-web: `.route("path", web::get().to(handler))` + `#[get("path")]` attributes
10. Rocket: `#[get("path")]` attributes + `.mount("prefix", routes![...])`
11. ASP.NET Core: `[HttpGet]`, `[Route]` attributes + minimal API `app.Map*`
12. Vapor: `app.get("seg", "seg", use: handler)` + grouped routes

### Phase 6: PHP CMS frameworks (F26)

Implement `src/frameworks/detectors/symfony.ts`, `yii2.ts`, `yii3.ts`, `wordpress.ts`.

1. Symfony PHP attribute routes (`#[Route(...)]`) + class-level prefix
2. Symfony legacy docblock `@Route(...)` annotation support
3. Symfony YAML routing (`config/routes.yaml`) parser
4. Symfony PHP routing (`$routes->add(...)`) call extraction
5. Symfony `EventSubscriberInterface::getSubscribedEvents()` → `hook` edges
6. Yii2 URL rules array from config (`'urlManager' => ['rules' => [...]]`) + `yii\rest\UrlRule` CRUD expansion
7. Yii2 `actionXxx` method convention → route edge with default-convention path
8. Yii2 `behaviors()` returning `AccessControl`/`VerbFilter` → `middleware` edges
9. Yii3 `Route::get/post/...()->action([Class::class, 'method'])` + group prefix
10. WordPress `add_action` / `add_filter` → `hook` edges with priority/args metadata
11. WordPress `add_shortcode` → `hook` edges with `hookType: 'shortcode'`
12. WordPress `register_rest_route` → `route` edges with full `/wp-json/` prefix
13. WordPress `register_post_type` / `register_taxonomy` → `hook` edges
14. WordPress template hierarchy file scan → file-based route edges
15. `mapx hooks` CLI command + `mapx_hooks` MCP tool

### Phase 7: Documentation + F19 integration

1. Update `docs/cli-reference.md` with `mapx routes` command
2. Update `docs/mcp-integration.md` with `mapx_routes` tool
3. Add `docs/framework-support.md` with detection signals and supported patterns per framework
4. Integrate route edges into F19 smart context: `mapx_context` now includes nearby route edges when querying controller/handler files
5. Update `docs/getting-started.md` with framework detection section

## Risk register

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|-----------|
| Regex-based extraction misses edge cases in complex route files | Medium | Low | Emit with `confidence: "inferred"` so users know extraction is approximate |
| Go/Rust/Java/C#/Swift parsers (F20) not yet implemented when work starts | High | High | F21–F25 do not require AST-level parsing — all use regex; can proceed before I12 merges for Python/JS/TS/PHP/Ruby |
| Next.js App Router + Pages Router coexist in same project | Medium | Low | Detect both directories independently; emit routes from whichever exists |
| NestJS path prefix accumulation across `@Module` imports | Medium | Medium | Only resolve prefix from `@Controller()` decorator in same file; cross-module prefix resolution deferred to future iteration |
| Drupal hook name collision (two modules implementing same hook) | Low | Low | Both emit separate `hook` edges — no deduplication needed |
| WordPress `add_action` callback as anonymous function | Medium | Low | Emit `hook` edge to the containing file as target; anonymous body not separately symbolised |
| Symfony YAML `resource:` import triggers recursive directory scan | Medium | Medium | Cap recursion depth; defer nested import resolution to future iteration |
| Yii2 URL rules split across multiple config files | Medium | Medium | Only scan the primary `config/web.php` or `config/main.php`; other config files flagged as `confidence: "inferred"` |
