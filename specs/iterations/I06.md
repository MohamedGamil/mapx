## Objective

Complete the Laravel-specific graph layer by resolving the two remaining architectural patterns:

1. **Facade calls** — rewrite static `Cache::`, `DB::`, `Log::` edges from opaque facade stubs to their actual underlying service classes
2. **Event/Job/Notification dispatch** — extract `dispatch` and `notify` edges from the various PHP forms used in Laravel to queue and emit async work units

After I06, the mapx PHP/Laravel graph is semantically complete for the application layer. The graph will correctly represent all major dependency types: imports, type hints, service bindings, routes, relationships, facade calls, and async dispatch.

## F11 Summary: Facade Resolution

**What it adds**: Post-parse edge rewriting step that consults a built-in facade map (and optionally a user-defined map in config) to rewrite `call` edges targeting known facade aliases to their concrete service FQN.

**Key design**: Facade resolution is `inferred` (not `verified`) because facade binding is a Laravel runtime concern, not statically determinable from source code alone. The facade alias is preserved in `metadata.facadeAlias` for traceability.

**Interaction with other features**:
- `Route::` facade calls: excluded from F11 rewriting — already consumed by F08 as `route` edges
- `Schema::` in migrations: suppressed by F10 exclusions before reaching F11
- `DB::` facade: marked with `metadata.isRawDbAccess = true` as a signal for raw database access analysis

**Configuration**: Users can extend the built-in map with custom facades in `.mapx/config.json`.

---

## F12 Summary: Event/Job/Notification Dispatch

**What it adds**: Two new edge types (`dispatch` and `notify`) capturing async work dispatch patterns. Dispatch targets (event/job/notification classes) receive `laravelRole` metadata. Listener classes receive `laravelRole = 'listener'`.

**Dispatch forms covered**:
- `event(new X())` — global helper
- `Event::dispatch(new X())` — facade form
- `X::dispatch()` — Dispatchable trait form
- `dispatch(new X())` — helper function (jobs)
- `$user->notify(new X())` — notifiable model method
- `Notification::send($users, new X())` — facade form

**Roles added**: `event`, `job`, `notification`, `listener`

**Stretch goal**: `$listen` array in `EventServiceProvider` — maps event classes to listener classes. Tree-sitter query complexity may require post-parse array literal parsing. May slip to I07 if blocked.

## Acceptance Criteria (I06 complete when all pass)

### F11 — Facade Resolution

- [ ] `Cache::get(...)` → edge to `Illuminate\Cache\CacheManager`, `verifiability = 'inferred'`
- [ ] `DB::table(...)` → edge to `Illuminate\Database\DatabaseManager`, `isRawDbAccess = true`
- [ ] `Log::info(...)` → edge to `Illuminate\Log\LogManager`
- [ ] `Mail::send(...)` → edge to `Illuminate\Mail\Mailer`
- [ ] `Route::get(...)` NOT rewritten by F11 (handled by F08)
- [ ] Custom facade in `.mapx/config.json` overrides built-in map entry
- [ ] `metadata.facadeAlias` preserved on rewritten edges
- [ ] All 26 built-in framework facades correctly mapped (see F11 spec for full list)

### F12 — Event/Job/Notification Dispatch

- [ ] `event(new UserRegistered($user))` → `dispatch` edge, `verified`, `dispatchedRole = 'event'`
- [ ] `ProcessOrder::dispatch($order)` → `dispatch` edge, `dispatchMethod = 'dispatch'`
- [ ] `dispatch(new ProcessOrder($order))` → `dispatch` edge, `dispatchedRole = 'job'`
- [ ] `$user->notify(new InvoiceReady($invoice))` → `notify` edge
- [ ] `Notification::send($users, new InvoiceReady(...))` → `notify` edge
- [ ] Event classes → `laravelRole = 'event'`
- [ ] Job classes → `laravelRole = 'job'`
- [ ] Notification classes → `laravelRole = 'notification'`
- [ ] Listener classes → `laravelRole = 'listener'`
- [ ] `dispatch` edges rendered as dashed arrows in DOT/SVG exporter
- [ ] `notify` edges rendered as dotted arrows in DOT/SVG exporter

### Common

- [ ] TypeScript type-check: `npx tsc --noEmit` passes with 0 errors
- [ ] All I04 and I05 acceptance criteria continue to pass (no regressions)

## Notes

- F11 and F12 are independent of each other within I06 and can be developed in parallel
- F11 is a post-parse **edge rewriting step**, not a new parser query — it runs after tree-sitter parsing, before storing to the graph
- F12 adds tree-sitter queries; consider whether `dispatch` and `notify` edge types need dedicated colour/style in the DOT exporter vs. reusing existing styles
- After I06, the PHP/Laravel graph is feature-complete for the core architecture. Future iterations (I07+) can address: Blade templates, config key references, named route resolution, Artisan commands, mail dispatch, and observer patterns
