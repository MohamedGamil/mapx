## Objective

Fix mapx's fundamental assumption that a project is always a single git repository. Real projects have git submodules, monorepo-style sibling repos, and VS Code multi-root workspaces. Without F18, incremental scans for submodules are incorrect (the parent's `git ls-tree` cannot enumerate per-file hashes inside submodule directories), cross-repo dependencies are invisible, and unregistered nested repos are silently indexed with wrong tracking semantics.

After I10, mapx:
- Auto-discovers `.gitmodules` and registers each submodule as an independent tracked repo
- Detects VS Code `.code-workspace` folders and sibling git repos
- Performs per-repo incremental scans using each repo's own git index
- Tracks cross-repo dependency edges
- Exposes `mapx workspaces` CLI and `mapx_workspaces` MCP tool

## F18 Summary

### New capability: `WorkspaceManager`

New `src/core/workspace-manager.ts` class that:
1. Parses `.gitmodules` → `SubmoduleInfo[]`
2. Scans sibling directories (opt-in) → `SubmoduleInfo[]`
3. Reads `*.code-workspace` JSON → `SubmoduleInfo[]`
4. Detects unregistered nested `.git` during file walk
5. Converts discovered repos to `RepoConfig[]` entries and writes to config

### Scanner: per-repo isolation

The scanner (`src/core/scanner.ts`) is updated to iterate `config.repos[]` independently, calling `getGitBlobHashes()` with each repo's own root as `cwd`. This fixes incremental scan accuracy for submodules.

### Nested `.git` protection

During `discoverFiles()`, when a directory contains `.git` and is neither the current `repoRoot` nor a registered submodule path, the scanner stops walking that directory and emits a warning. Files inside are excluded.

### Cross-repo edges

`GraphEdge.targetRepo` field added. When symbol resolution determines an import resolves to a file in a different registered repo, `target_repo` is populated in the `edges` table. Schema migration adds this nullable column.

### New CLI surface

| Command | Description |
|---------|-------------|
| `mapx workspaces` | List registered repos with scan status |
| `mapx workspaces discover` | Auto-discover submodules + VS Code workspace repos |
| `mapx workspaces add <path>` | Manually register a repo |
| `mapx workspaces remove <name>` | Unregister a repo |
| `mapx scan --all` | Scan all registered repos |
| `mapx update --all` | Incremental update all repos |
| `mapx status --all` | Per-repo status table |
| `mapx export --all` | Multi-repo combined LLM export |
| `mapx export --repo=<name>` | Single-repo export |

### New MCP tool

`mapx_workspaces` — returns workspace info, registered repos, scan status, cross-repo edge counts.

## Key design decisions

### Config-driven, not filesystem-driven at scan time

Discovered repos are **registered in `.mapx/config.json`** before any scan. The scanner only processes repos listed in `config.repos[]`. This means:
- Discovery is a one-time interactive step (during `init` or `workspaces discover`)
- Scans are deterministic and repeatable
- The user has explicit control over what gets scanned

If a user wants to add a new submodule mid-project, they run `mapx workspaces discover` or `mapx workspaces add`.

### Uninitialised submodules: warn but don't fail

Submodules listed in `.gitmodules` but not yet initialised (no `.git` entry in their directory) are reported but not registered. Attempting to scan an uninitialised submodule would produce empty results and confuse the user. The warning includes the `git submodule update --init` command.

### Peer repo discovery is opt-in

Discovering sibling directories is inherently imprecise — a developer's home directory might have dozens of unrelated git repos as siblings. Peer discovery is shown during `mapx workspaces discover` only with `--include-peers` or an explicit prompt choice. VS Code `.code-workspace` discovery is opt-in by default (auto-detected if the file is present, but user confirms before registering).

### Alignment with Canonical Schema Sequence

Following the canonical schema sequence defined in ADR-003:
- Schema v4: F14 cluster tables (I08)
- Schema v5: F18 `edges.target_repo` column (I10)

This establishes a clear migration order where I10 depends on the schema changes from I08.

## Notes

- The `discoverSubmodules()` helper should handle both the `path = ...` and `url = ...` styles in `.gitmodules`, including relative URLs (`../sibling.git`).
- When a submodule's path exists but contains a `.git` file (text file, not directory) — this is a normal git submodule; treat it as initialised.
- When a submodule's path exists and contains a `.git` directory — this is a detached-worktree submodule setup; also initialised.
- `git submodule foreach --quiet --recursive 'echo $displaypath'` is an alternative to parsing `.gitmodules` directly; however, this requires git to be installed and the submodules to be initialised. Parsing `.gitmodules` directly is more robust (works even for uninitialised submodules).
