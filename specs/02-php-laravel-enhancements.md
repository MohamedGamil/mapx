# PHP & Laravel Support Enhancements — Backlog

This document is the overview and prioritised backlog for improving mapx's understanding of PHP codebases, with particular emphasis on the **Laravel framework** and its architectural conventions.

It accompanies the feature-level specs in `specs/features/F05` through `F12` and the iteration plans in `specs/iterations/I04` through `I06`.

---

## Current State

The PHP parser (`src/parsers/languages/php.ts` + `queries/php/`) currently extracts:

| Capability | Status |
|------------|--------|
| Class / interface / trait / enum declarations | ✅ |
| Method declarations | ✅ |
| Function declarations | ✅ |
| Properties and class constants | ✅ |
| Namespace `use` imports | ✅ partial — captures only the final `name` segment, losing the FQN |
| `extends` / `implements` relationships | ✅ |
| `new ClassName()` instantiations | ✅ |
| Static / member call edges | ✅ but treated as generic `call` — no semantic distinction |
| Constructor injection / type hints | ❌ not extracted |
| Return type declarations | ❌ not extracted |
| Eloquent relationships | ❌ not extracted |
| Route-to-controller bindings | ❌ not extracted |
| Service container bindings | ❌ not extracted |
| Facade static calls | ❌ not semantically resolved |
| Event / Job dispatch | ❌ not extracted |
| Artisan command signatures | ❌ not extracted |
| Laravel-specific noise exclusion | ❌ migrations/seeds/storage not suppressed |

## Problem Areas

### 1. Incomplete namespace resolution

`use App\Services\UserService;` is captured as `UserService` rather than `App\Services\UserService`. Symbol resolution across files is therefore unreliable — two classes named `UserService` in different namespaces are indistinguishable in the graph.

### 2. Missing type-hint dependencies

Laravel's primary dependency wiring mechanism is constructor injection:

```php
public function __construct(
    private readonly UserService $userService,
    private readonly MailService $mailService,
) {}
```

None of these create edges today. They are the single most important source of **verified** structural dependencies in any Laravel application.

### 3. Invisible Eloquent relationships

```php
class User extends Model {
    public function posts(): HasMany { return $this->hasMany(Post::class); }
    public function team(): BelongsTo { return $this->belongsTo(Team::class); }
}
```

`hasMany(Post::class)` and `belongsTo(Team::class)` are high-value semantic edges — they express the data model. They are invisible today.

### 4. Route files generate noise

`routes/web.php` and `routes/api.php` contain hundreds of `Route::get(...)` / `Route::post(...)` calls. Without special treatment these appear as a dense tangle of generic `call` edges to every controller in the app, making the graph misleading.

### 5. Service provider bindings are opaque

```php
$this->app->bind(UserRepositoryInterface::class, UserRepository::class);
$this->app->singleton(CacheService::class, fn($app) => new CacheService($app->make(Redis::class)));
```

These are the IoC container's wiring points. Without parsing them, the graph cannot represent the abstract-to-concrete mapping that governs runtime composition.

### 6. Facades look like noise

```php
Cache::get('user:42');
DB::table('orders')->where(...)->get();
Mail::to($user)->send(new WelcomeMail());
```

Each of these resolves at runtime to a real service class. Without a facade-to-class map, they appear as unresolvable static calls.

### 7. Events, Jobs, and Notifications create invisible cross-cutting edges

```php
event(new UserRegistered($user));
ProcessOrder::dispatch($order);
Notification::send($users, new InvoiceReady($invoice));
```

These are the async and event-driven seams of the application. They are currently invisible.

### 8. Scan noise: migrations, seeds, compiled assets

- `database/migrations/` — pure schema DDL, no cross-file PHP dependencies
- `database/seeders/` — data fixtures, rarely referenced from app code
- `storage/` — runtime-generated files, should never be scanned
- `bootstrap/cache/` — compiled container/route cache, generated by artisan
- `public/` — web-accessible assets, not application logic
- `resources/views/` — Blade templates (PHP-adjacent but not parsed as PHP today)

Without exclusions these directories inflate file counts and add meaningless edges.

## Design Principles for PHP/Laravel Enhancements

### Verified vs. inferred (ties into F01)

The verifiability model from F01 is especially important in PHP:

| Pattern | Verifiability |
|---------|--------------|
| `use App\Services\UserService;` + typed parameter | `verified` |
| `new UserService()` where class is `use`-imported | `verified` |
| `$this->hasMany(Post::class)` | `verified` (class constant — statically known) |
| `Route::get('/users', [UserController::class, 'index'])` | `verified` (array syntax) |
| `Route::get('/users', 'UserController@index')` | `inferred` (string — resolved at runtime) |
| `$this->userService->save()` (typed via constructor) | `verified` |
| `$this->userService->save()` (untyped / injected via magic) | `inferred` |
| Facade calls (`Cache::get()`) | `inferred` unless facade map is loaded |

### Namespace-first resolution

All symbol resolution should be namespace-aware. The `use` import table for each file must be built before edges are created so that short names (`UserService`) resolve to FQNs (`App\Services\UserService`).

### Do not over-parse Laravel internals

The goal is to understand **application code**, not the framework itself. Edges into `Illuminate\*`, `vendor\*`, and `Laravel\*` namespaces should be recorded (for completeness) but flagged as `external` in a future metadata field so exporters can optionally suppress them.

### Symbols carry Laravel-role metadata

Where a class can be identified as a specific Laravel construct, record this in `ExtractedSymbol.metadata`:

```typescript
metadata: {
  laravelRole: 'controller' | 'model' | 'service_provider' | 'middleware' |
               'job' | 'event' | 'listener' | 'observer' | 'command' |
               'request' | 'resource' | 'policy' | 'factory' | 'seeder' |
               'migration' | 'notification' | 'mail' | 'cast' | 'rule'
}
```

This allows exporters, metrics, and future tooling to filter or annotate by architectural role.

## Noise Budget

The table below estimates the proportion of edges that are currently noise (inferred / unresolvable) in a typical mid-size Laravel application, and the expected state after these enhancements:

| Edge category | Before | After I04+I05+I06 |
|---------------|--------|---------------------|
| `use` imports (FQN) | 40% lost (short name only) | 100% resolved |
| Constructor injection | 0% captured | ~80% captured (typed params) |
| Eloquent relationships | 0% captured | ~90% captured (`::class` form) |
| Route-to-controller | 100% noise (`call` edges) | verified for array syntax |
| Service bindings | 0% captured | ~70% captured |
| Facade calls | 100% noise | ~60% semantically mapped (I06) |
| Event/Job dispatch | 0% captured | ~85% captured (`::class` form) |

## Reference: Laravel Directory Structure

The standard Laravel directory layout and the mapx treatment after these enhancements:

```
app/
├── Console/
│   └── Commands/          → scan; extract Artisan command signatures as metadata
├── Exceptions/            → scan; handler class
├── Http/
│   ├── Controllers/       → scan; detect controller role, route bindings (I05/F08)
│   ├── Middleware/        → scan; detect middleware role
│   └── Requests/          → scan; FormRequest subclasses
├── Jobs/                  → scan; detect job role, dispatch edges (I06/F12)
├── Listeners/             → scan; detect listener role, event bindings
├── Mail/                  → scan; detect mailable role
├── Models/                → scan; detect model role, Eloquent relationships (I05/F07)
├── Notifications/         → scan; detect notification role
├── Observers/             → scan; detect observer role
├── Policies/              → scan; detect policy role
├── Providers/             → scan; service container bindings (I05/F09)
├── Services/              → scan; general service classes
└── Events/                → scan; detect event role, listener edges (I06/F12)

bootstrap/
└── cache/                 → EXCLUDE (generated)

config/                    → scan (future: config key references)

database/
├── migrations/            → EXCLUDE (DDL only, no app logic)
├── seeders/               → EXCLUDE (data fixtures)
└── factories/             → EXCLUDE (test data, low-value for prod graph)

public/                    → EXCLUDE (web assets)

resources/
└── views/                 → EXCLUDE (Blade templates — separate future feature)

routes/
├── web.php                → scan with route-parsing mode (I05/F08)
├── api.php                → scan with route-parsing mode (I05/F08)
└── console.php            → scan with route-parsing mode

storage/                   → EXCLUDE (runtime-generated)

tests/                     → configurable; excluded by default (see F03)

vendor/                    → EXCLUDE (always)
```
